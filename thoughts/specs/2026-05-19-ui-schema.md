# Formspec Experience, Operations, and Trace Graph Specification

**Version:** 1.0.0-draft.1
**Date:** 2026-05-19
**Status:** Draft companion specification
**Short names:** FEX, FOP, FTG
**Depends on:** Formspec Core, Component, Theme, Mapping, References, Ontology, Registry, Validation Report, Respondent Ledger, Intake Handoff, Verification Receipt, Posture Declaration, and Conformance Suite specifications.

**Introduces:** `formspec-foundations` — a shared schema package containing cross-cutting primitives (`DefinitionRef`, `TypedRef`, `HashString`, `Ref`). This specification both defines those primitives inline (§15.4) and consumes them from every artifact it adds or modifies. Existing companion specs (Component, Locale, IntakeHandoff, Respondent Ledger) currently inline equivalents; extracting them into a foundations package is the recommended migration and is gated on this spec landing.

---

## Abstract

This specification defines three additive Formspec companion artifacts:

```text
Experience Document   → actor/task/abstract interaction model
Operations Document   → runtime action/submission/lifecycle policy
Trace Graph Document  → generated cross-artifact relationship index
```

Together, these artifacts close the remaining practical parity gaps with XForms, XIML, and UsiXML/CAMELEON without duplicating existing Formspec schemas.

Formspec Core already defines the executable form model: Definition, Items, Binds, Shapes, Variables, Instances, Response pinning, validation, mapping, semantic registries, references, components, themes, ledgers, handoffs, and runtime evidence. The Component specification already defines a parallel presentation tree bound to Definition items and explicitly states that Component trees control layout and widget selection but cannot override core behavioral semantics.  The References specification already defines resource/context bindings to targets, including documentation, policy, regulation, knowledge bases, tools, APIs, and agent/human context.  The Respondent Ledger, Validation Report, Intake Handoff, Mapping, Registry, and related schemas already own durable event history, validation evidence, workflow handoff, data transformation, and semantic extension governance.

This specification therefore does **not** define new schemas for validation reports, lifecycle event storage, workflow handoff, signature verification, data projection, semantic concepts, design tokens, or authoring commands. Those concerns are already owned by existing Formspec artifacts.

# 1. Status of This Document

This document is a draft companion specification. It has not been submitted to any standards body. Implementors are encouraged to experiment with this specification while Formspec remains greenfield.

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** are to be interpreted as described in RFC 2119 and RFC 8174.

# 2. Design Principles

## 2.1 Single Ownership Per Concern

Each artifact MUST own one primary concern. A conforming processor MUST NOT require authors to duplicate the same semantic fact in multiple source artifacts.

For example:

```text
Component owns component-to-item binding.
Experience owns task/unit grouping.
Operations owns runtime action/submission policy.
Mapping owns external payload transformation.
Ontology owns Definition-specific semantic bindings.
References owns contextual resources.
Trace Graph indexes relationships derived from these artifacts.
```

If a generated Trace Graph edge conflicts with a source artifact, the source artifact is authoritative.

## 2.2 Source Artifacts vs Result Artifacts

Formspec artifacts fall into three categories:

```text
Source artifacts:
  Definition, Experience, Operations, Component, Theme, Registry,
  Ontology, References, Mapping, Posture Declaration

Runtime/evidence artifacts:
  Response, ValidationReport, ValidationResult, RespondentLedger,
  RespondentLedgerEvent, IntakeHandoff, VerificationReceipt,
  Determination, Changelog

Generated/index artifacts:
  TraceGraph
```

The Trace Graph is a generated result/index artifact. It MUST NOT be required for normal form execution.

## 2.3 Closed Schemas and Named Extension Seams

Schemas SHOULD remain closed by default. Extension seams SHOULD use `extensions` objects with `x-`-prefixed keys governed by the Registry mechanism. This aligns with the existing closed-schema posture described in ADR 0056.

# 3. Relationship to Prior Art

## 3.1 XForms

This specification addresses the remaining XForms-style operation layer:

```text
runtime actions
submission policy
validation-mode selection
transaction semantics
runtime lifecycle emission
durable ledger event mapping
```

Formspec Core already covers the model side: item tree, instance data, Binds/MIPs, validation, repeat groups, non-relevant behavior, and reactive processing. The Experience/Operations pair adds the missing runtime action and submission semantics.

## 3.2 UsiXML / CAMELEON

This specification addresses model-based UI abstraction by introducing Experience Documents with:

```text
actors
tasks
abstract interaction units
applicability/context hints
```

These concepts correspond to task/user/context and Abstract UI layers, while existing Component and Theme documents remain the Concrete UI layer.

## 3.3 XIML

This specification addresses interaction-data traceability through the Trace Graph. The Trace Graph indexes relationships among tasks, units, actions, items, components, concepts, references, validation results, mappings, ledger events, handoffs, determinations, changelogs, and receipts.

The Trace Graph is generated from source and evidence artifacts; it is not a duplicate authored model.

# 4. Artifact Overview

This specification defines three artifacts.

```text
experience.schema.json
operations.schema.json
trace-graph.schema.json
```

It also introduces a shared foundations package and defines small changes to existing artifacts:

```text
foundations.schema.json     → DefinitionRef, TypedRef, HashString, Ref ($defs only)
component.schema.json       → unitRef, taskRefs, actionRef, conceptRefs
references.schema.json      → typed targets (consumes foundations TypedRef)
locale.schema.json          → actor/task/unit/action/submission string keys
mapping.schema.json         → non-relevant-field behavior per targetSchema descriptor
conformance-suite.schema    → Experience, Operations, and Trace test kinds
```

`foundations.schema.json` is a `$defs`-only package; it carries no top-level document. Consumers `$ref` its primitives. The Mapping modification (per-`targetSchema` `nonRelevantBehavior`) replaces the earlier draft's `Submission.nonRelevantBehavior` — boundary-translation policy belongs in the boundary-translation layer.

A future `package.schema.json` MAY be added for artifact bundling and variant resolution, but package semantics are outside this specification.

# 5. Experience Document

## 5.1 Purpose

An Experience Document defines the abstract journey through a Formspec Definition.

It answers:

```text
Who is using this form?
What tasks are they completing?
Which Definition items and semantic concepts belong together?
Which runtime actions are offered in each abstract interaction unit?
What actor/platform/context does this experience apply to?
```

An Experience Document MUST NOT define field structure, validation rules, calculation rules, submission payload transformations, durable events, or concrete component layout.

## 5.2 Document Shape

```json
{
  "$formspecExperience": "1.0",
  "url": "https://example.gov/forms/intake/experiences/public-web",
  "version": "1.0.0",
  "targetDefinition": {
    "url": "https://example.gov/forms/intake",
    "compatibleVersions": ">=1.0.0 <2.0.0"
  },
  "applicability": {
    "actors": ["respondent"],
    "platforms": ["web"],
    "contexts": ["public-intake"],
    "locales": ["en-US"],
    "posture": "https://agency.gov/postures/public-intake|1.0.0"
  },
  "actors": {},
  "tasks": {},
  "units": []
}
```

Richer context-of-use adaptation axes — disability preferences, device modality, offline state, kiosk mode, legal jurisdiction, authentication context — belong in future Package variants, not in Experience. This specification deliberately keeps `applicability` small and defers a `context.schema.json` until Package semantics exist.

## 5.3 Required Properties

An Experience Document MUST include:

```text
$formspecExperience
url
version
targetDefinition
```

An Experience Document SHOULD include at least one `unit`.

## 5.4 Actors

Actors describe participant roles in the experience.

```json
{
  "actors": {
    "respondent": {
      "label": "Applicant",
      "description": "The person completing the public intake."
    },
    "caseworker": {
      "label": "Caseworker",
      "description": "A staff member assisting or reviewing the intake."
    }
  }
}
```

A processor MUST treat actor identifiers as stable within the Experience Document. Actor labels MUST NOT be interpreted as authentication or authorization policy. Authorization is owned by the host: a host MAY consult Posture Declaration, the WOS governance overlay, or any other policy mechanism to decide action eligibility. Formspec is authz-agnostic; this specification defines what an actor sees and can do, not who is permitted to be an actor.

## 5.5 Tasks

Tasks describe user goals or work activities.

```json
{
  "tasks": {
    "identifyApplicant": {
      "label": "Identify the applicant",
      "actor": "respondent",
      "intent": "establish_identity",
      "outcome": "identity_available"
    },
    "provideContact": {
      "label": "Provide contact information",
      "actor": "respondent",
      "intent": "enable_followup",
      "outcome": "contact_channel_available"
    }
  }
}
```

Tasks MAY be hierarchical:

```json
{
  "tasks": {
    "completeApplication": {
      "label": "Complete application",
      "children": [
        "identifyApplicant",
        "provideContact",
        "certifyAndSubmit"
      ]
    }
  }
}
```

If a task references an actor, that actor SHOULD exist in `actors`.

## 5.6 Abstract Interaction Units

A Unit is an abstract interaction unit: a task-oriented grouping of Definition items, semantic concepts, references, and runtime actions.

```json
{
  "units": [
    {
      "id": "identity",
      "kind": "data-entry",
      "taskRefs": ["identifyApplicant"],
      "itemRefs": ["applicantName", "dateOfBirth"],
      "conceptRefs": ["person.fullName", "person.birthDate"],
      "actionRefs": ["saveDraft", "continue"],
      "referenceRefs": ["identity-policy"],
      "accessibility": {
        "label": "Applicant identity",
        "description": "Information used to identify the applicant."
      }
    }
  ]
}
```

## 5.7 Unit Kinds

The following Unit kinds are defined:

```text
data-entry
guidance
review
summary
confirmation
navigation
decision
routing
evidence-upload
signature
error-resolution
```

Custom Unit kinds MUST be prefixed with `x-` and governed by Registry entries. Speculative kinds — including agent-collaboration and external-system-handoff variants — MUST start their lives as `x-` extensions and earn promotion into the closed enum via conformance fixtures.

## 5.8 Unit Semantics

An Experience Processor:

1. MUST validate that `itemRefs` resolve to Definition item paths when the target Definition is available.
2. SHOULD validate that `conceptRefs` resolve through Ontology or Registry when those artifacts are available.
3. SHOULD validate that `actionRefs` resolve to Operations actions when an Operations Document is available.
4. MUST NOT treat Unit grouping as equivalent to Definition item structure.
5. MUST NOT change validation, relevance, requiredness, readonly, or calculation semantics based on Units alone.
6. SHOULD preserve Unit identity for Component binding and Trace Graph generation.

# 6. Operations Document

## 6.1 Purpose

An Operations Document defines runtime actions, submissions, validation-mode selection, lifecycle emission policy, durable ledger event mapping, and handoff production policy.

It answers:

```text
What can an actor do?
When can they do it?
What validation mode applies?
What runtime effects occur?
What submission runs?
What evidence artifacts are produced?
What durable ledger events are recorded?
```

Operations are respondent/runtime/workflow actions. They are distinct from authoring commands, which are already covered by the Core Command Catalog.

## 6.2 Document Shape

```json
{
  "$formspecOperations": "1.0",
  "url": "https://example.gov/forms/intake/operations/public-web",
  "version": "1.0.0",
  "targetDefinition": {
    "url": "https://example.gov/forms/intake",
    "compatibleVersions": ">=1.0.0 <2.0.0"
  },
  "applicability": {
    "experienceRefs": [
      "https://example.gov/forms/intake/experiences/public-web|1.0.0"
    ]
  },
  "actions": {},
  "submissions": {},
  "lifecycle": {}
}
```

## 6.3 Required Properties

An Operations Document MUST include:

```text
$formspecOperations
url
version
targetDefinition
```

An Operations Document SHOULD include at least one `action`.

## 6.4 Actions

An Action is a runtime operation.

```json
{
  "actions": {
    "submitApplication": {
      "label": "Submit application",
      "description": "Submits the completed application for review.",
      "actor": "respondent",
      "intent": "finalize_response",
      "uiValidationMode": "submit",
      "preconditions": [
        "valid(#)",
        "$certificationAccepted = true"
      ],
      "confirmation": {
        "required": true,
        "title": "Submit application?",
        "message": "After submission, changes may require an amendment."
      },
      "requiredPosture": "https://agency.gov/postures/public-intake|1.0.0",
      "effects": [
        {
          "type": "runSubmission",
          "submission": "finalApplication"
        },
        {
          "type": "recordLedgerEvent",
          "eventType": "response.completed"
        }
      ]
    }
  }
}
```

### 6.4.1 Posture as an Admission Constraint, Not a Precedence

`Action.requiredPosture` declares the posture floor an Action requires to be invoked. `Experience.applicability.posture` declares the posture context an Experience was designed for. These are not in precedence — they describe a constraint relationship: the deployment's currently-admitted Posture Declaration MUST satisfy the Action's `requiredPosture` for the Action to be invokable. A processor MUST reject Action invocations whose `requiredPosture` is not admitted by the resolved deployment posture, even when `Experience.applicability.posture` matched.

When an Action's effects trigger signature capture or verification, the resolved Posture's `signaturePolicy.allowedSigningIntents` MUST admit the operative intent. Admission failure produces `SignatureAdmissionFailed` per ADR 0090 D8.

## 6.5 UI Validation Mode

An Action MAY declare one of the following UI validation modes:

```text
none
continuous
submit
demand
```

`uiValidationMode` governs what the respondent sees while the Action is being performed — inline error rendering, validation-on-blur, validation-at-submit, on-demand revalidation. It is distinct from `Submission.boundaryValidationMode` (§6.9), which governs the validation pass at the external boundary. The two run independently and serve different audiences; processors MUST run both passes when an Action invokes a Submission.

If omitted, processors MUST default to:

```text
continuous   for non-submitting actions
submit       for actions that run a submission
```

Validation evidence is represented by `ValidationReport` and `ValidationResult`, not by Operations. The Validation Report schema already defines validity, findings, counts, timestamps, and validation invariants.

## 6.6 Preconditions

Preconditions are FEL expressions evaluated before effects.

If any precondition evaluates to false or null, a processor MUST NOT apply effects.

## 6.7 Confirmation

An Action MAY require confirmation.

```json
{
  "confirmation": {
    "required": true,
    "title": "Submit application?",
    "message": "After submission, changes may require an amendment.",
    "confirmLabel": "Submit",
    "cancelLabel": "Cancel"
  }
}
```

If `confirmation.required` is true, a processor MUST obtain or verify confirmation evidence before applying effects.

## 6.8 Effects

Built-in effect types include:

```text
setValue
addRepeatInstance
removeRepeatInstance
runSubmission
recordLedgerEvent
produceIntakeHandoff
requireVerificationReceipt
emitRuntimeEvent
setResponseStatus
```

Custom effect types MUST be prefixed with `x-` or governed by Registry entries.

Effects MUST execute in declared order within a single transaction.

### 6.8.1 Effect Authority Boundaries

The following normative boundaries apply to built-in effects. Operations orchestrates; it does not own these artifact structures.

- `recordLedgerEvent` MUST reference a ledger event type only. It MUST NOT embed the event body, integrity profile, embedded event stream, checkpoint, offline-authoring envelope, or changelog content. Those structures are owned by the Respondent Ledger schema. `eventType` MUST be either a value from the Respondent Ledger `EventType` enum (defined in `respondent-ledger-event.schema.json` `$defs.EventType`) or an `x-`-prefixed extension type governed by Registry. Processors MUST reject `eventType` values that are neither.
- `produceIntakeHandoff` MUST NOT redefine the Intake Handoff envelope, validated boundary semantics, or signature/receipt linkage. The Intake Handoff schema owns the validated intake-to-workflow boundary.
- `requireVerificationReceipt` MUST NOT redefine signature policy or verification result structure. Posture Declaration owns signature/custody policy; Verification Receipt owns verification evidence.

Effects producing IntakeHandoff or recording chained Ledger events cross the Formspec ↔ WOS/Trellis seam. Processors MUST honor the Trellis envelope obligations (`eventHash`/`priorEventHash` per Respondent Ledger §6.2) and the workflow host's custody hook for governed-case identity (Intake Handoff §13).

A processor MUST reject Operations Documents whose effects embed body content owned by another schema.

## 6.9 Submissions

A Submission defines runtime submission policy. Submissions MUST NOT define custom payload transformation logic; any Response-to-external transformation MUST be expressed through a Mapping Document referenced via `mappingRef`. Submissions MUST NOT redefine the Intake Handoff envelope, the Validation Report structure, signature/custody policy, or non-relevant-field handling — those are owned by their respective schemas (Intake Handoff, Validation Report, Posture Declaration, Verification Receipt, Mapping).

```json
{
  "submissions": {
    "finalApplication": {
      "label": "Final application submission",
      "payload": {
        "kind": "mappingRef",
        "mappingRef": "https://example.gov/mappings/intake-to-case|1.0.0"
      },
      "boundaryValidationMode": "submit",
      "outputs": [
        {
          "type": "validationReport"
        },
        {
          "type": "intakeHandoff",
          "initiationMode": "publicIntake"
        }
      ],
      "destination": {
        "type": "host",
        "uri": "urn:agency:intake:submit"
      }
    }
  }
}
```

`boundaryValidationMode` governs the validation pass run at the external boundary, immediately before payload construction (§9.4). It is distinct from `Action.uiValidationMode` (§6.5); both passes run independently when an Action invokes a Submission. Boundary failure aborts the Submission per §9.4 step 3.

## 6.10 Payload Kinds

Supported payload kinds:

```text
none          → no payload; Submission emits only side effects
mappingRef    → payload built through a Mapping Document
```

`kind: mappingRef` MUST carry a `mappingRef` pointing to a Mapping Document. Identity projections (full canonical Response, data tree only, subtree at a path) are degenerate Mapping Documents with `autoMap: true` and an identity `targetSchema`. The spec deliberately does not provide sentinel sugar (`"response"`, `"data"`, `"path"`) in v1.0; if conformance fixtures show authors repeatedly hand-writing the same identity mappings, a future minor version MAY add the sugar.

Mapping already owns Response-to-external-schema transformation, including JSON, XML, CSV, value maps, coercion, conditions, arrays, reverse mapping, and per-`targetSchema` non-relevant-field handling.

## 6.11 Submission Outputs

Submissions MAY produce or reference:

```text
Response
ValidationReport
RespondentLedgerEvent
IntakeHandoff
VerificationReceipt
Determination
mappedPayload
```

The Intake Handoff schema already owns the validated intake-to-workflow boundary.

## 6.12 Lifecycle Policy

Lifecycle policy declares which processor-emitted ephemeral runtime events the document opts into.

```json
{
  "lifecycle": {
    "emit": [
      "action.started",
      "action.completed",
      "action.failed",
      "submission.ready",
      "submission.completed",
      "submission.failed"
    ]
  }
}
```

`lifecycle.emit` is a subscription filter — the processor automatically emits these events around action and submission invocation per §9.2, and the filter declares which ones the document wants surfaced to observers. It is not an authoring expression for new events.

This specification deliberately does not provide a `lifecycle.record` doc-wide list for durable ledger writes. Durable recording is expressed exclusively through per-Action `recordLedgerEvent` effects (§6.4, §6.8). One normative path prevents the precedence ambiguity that arises when the same event type appears in both places. If conformance fixtures show authors repeatedly declaring the same `recordLedgerEvent` effect on every action, a future minor version MAY add doc-wide sugar with a normative expansion algorithm.

Runtime events are ephemeral. Durable events are represented by Respondent Ledger artifacts. Respondent Ledger already owns append-only event history, integrity profiles, embedded events, checkpoints, offline authoring, and changelog modes.

# 7. Trace Graph Document

## 7.1 Purpose

A Trace Graph Document is a generated relationship index over Formspec artifacts.

It is not a normal authored source artifact. It is a materialized graph generated from source and evidence artifacts.

It answers:

```text
Which component renders which item?
Which unit collects which item?
Which action runs which submission?
Which submission uses which mapping?
Which handoff references which Response and ValidationReport?
Which reference explains or constrains which target?
Which ledger event records which runtime action?
```

## 7.2 Document Shape

```json
{
  "$formspecTraceGraph": "1.0",
  "url": "https://example.gov/forms/intake/traces/public-web",
  "version": "1.0.0",
  "targetDefinition": {
    "url": "https://example.gov/forms/intake",
    "version": "1.0.0"
  },
  "generatedAt": "2026-05-19T00:00:00Z",
  "generatedBy": {
    "processor": "formspec-trace-builder",
    "version": "0.1.0"
  },
  "inputs": [],
  "edges": []
}
```

### 7.2.1 Identity Model

A Trace Graph is identified by content digest, not by URL. Two Trace Graphs generated from the same input artifacts and the same builder version MUST produce the same content digest. `generatedAt` and `generatedBy` MUST be excluded from the digest scope; including them would defeat cache equivalence — the whole point of an index artifact.

`url` is OPTIONAL — a citation handle for hosted graphs. Local-only graphs (CI build artifacts, ad-hoc debugging dumps, AI impact-analysis traces) MAY omit it. `targetDefinition` MUST exact-pin (`url` + `version`); a Trace Graph indexes a specific Definition build, not a range.

## 7.3 Input Records

A Trace Graph MUST list each input artifact with `kind`, `uri`, `version`, and `digest`. A Trace Graph without complete input records is silently stale by construction and MUST NOT be treated as authoritative.

```json
{
  "inputs": [
    {
      "kind": "definition",
      "uri": "./definition.json",
      "version": "1.0.0",
      "digest": "sha256:..."
    },
    {
      "kind": "experience",
      "uri": "./experience.public-web.json",
      "version": "1.0.0",
      "digest": "sha256:..."
    },
    {
      "kind": "operations",
      "uri": "./operations.public-web.json",
      "version": "1.0.0",
      "digest": "sha256:..."
    },
    {
      "kind": "component",
      "uri": "./component.web.json",
      "version": "1.0.0",
      "digest": "sha256:..."
    }
  ]
}
```

A processor consuming a Trace Graph MUST recompute or compare digests against the named inputs before relying on edges. If any input digest does not match the current artifact, the Trace Graph is stale and edges MUST NOT override the source artifacts (see §7.6).

## 7.4 Edges

`from` and `to` are `TypedRef` values per `foundations.schema.json` `$defs.TypedRef` (§15.4) — strings of the form `<kind>:<id>` where `kind` is from the closed kind taxonomy (item, unit, task, action, component, submission, mapping, reference, concept, response, validationReport, ledgerEvent, handoff, verificationReceipt, determination, changelog) plus `x-`-prefixed extensions, and `id` follows the kind-specific identifier rules.

```json
{
  "from": "component:applicantNameInput",
  "rel": "renders",
  "to": "item:applicantName",
  "confidence": "derived",
  "source": {
    "artifact": "component",
    "path": "$.tree.children[0]"
  }
}
```

References typed targets (§8.2) consume the same `TypedRef` grammar. One parser, one conformance fixture set.

## 7.5 Confidence Values

Trace edges MUST declare confidence:

```text
derived
declared
inferred
observed
```

### 7.5.1 Derived

Mechanically generated from source artifacts.

Examples:

```text
component.bind              → component renders item
experience.unit.itemRefs    → unit collects item
operations.action.effects   → action runs submission
mapping.rules               → item maps-to external target
ontology.bindings           → item maps-to concept
references.target           → reference applies-to target
```

### 7.5.2 Observed

Generated from runtime evidence artifacts such as Ledger events, Responses, Validation Reports, Determinations, Handoffs, and Verification Receipts.

### 7.5.3 Inferred

Generated by tooling, AI, or heuristics. Inferred edges SHOULD require review before being treated as reliable.

### 7.5.4 Declared

Explicitly authored only when no existing artifact owns the relationship. Declared edges MUST NOT duplicate facts already expressible in source artifacts.

## 7.6 Authority Rule

A Trace Graph MUST NOT override source artifacts.

If a Trace Graph edge conflicts with an input artifact, the input artifact is authoritative and the Trace Graph is stale, invalid, or generated from different inputs.

## 7.7 Common Derived Edges

A Trace Builder SHOULD derive:

```text
component:<id> renders item:<path>
component:<id> triggers action:<id>
component:<id> realizes unit:<id>
unit:<id> collects item:<path>
unit:<id> offers action:<id>
unit:<id> supports-task task:<id>
action:<id> runs submission:<id>
submission:<id> uses mapping:<id>
submission:<id> produces handoff:<id>
reference:<id> applies-to target:<id>
item:<path> maps-to concept:<id>
mapping:<id> maps item:<path> to external:<path>
ledgerEvent:<id> records action:<id>
handoff:<id> references response:<id>
handoff:<id> references validationReport:<id>
verificationReceipt:<id> verifies signature:<id>
```

# 8. Modifications to Existing Schemas

## 8.1 Component Document Additions

Component nodes MAY include:

```text
unitRef
taskRefs
actionRef
conceptRefs
```

Example:

```json
{
  "id": "identityCard",
  "component": "Card",
  "unitRef": "identity",
  "taskRefs": ["identifyApplicant"],
  "conceptRefs": ["person.fullName"]
}
```

Action component:

```json
{
  "id": "submitButton",
  "component": "SubmitButton",
  "actionRef": "submitApplication"
}
```

Component remains the authoritative source for concrete UI layout and slot binding.

## 8.2 References Document Additions

References SHOULD NOT gain a general `relations[]` array.

Instead, References MAY accept `TypedRef` values (per `foundations.schema.json` `$defs.TypedRef`, shared with §7.4) in the `target` field, alongside the existing Bind-path dot-notation:

```text
item:applicantName
unit:identity
task:identifyApplicant
action:submitApplication
component:submitButton
submission:finalApplication
concept:person.fullName
```

This preserves the References specification's role as a resource/context binding layer while reusing the shared TypedRef grammar — no separate parser. The existing Bind-path form (e.g., `"applicantName"`, `"lineItems[*].amount"`, `"#"` for form-level) remains valid; it is the `item:` kind expressed as bare path.

## 8.3 Locale Document Additions

Locale SHOULD support the following namespaces:

```text
$actor.
$task.
$unit.
$action.
$submission.
```

Example:

```json
{
  "strings": {
    "$task.identifyApplicant.label": "Identify the applicant",
    "$unit.identity.label": "Applicant identity",
    "$action.submitApplication.label": "Submit application",
    "$action.submitApplication.confirmation.title": "Submit application?"
  }
}
```

## 8.4 Conformance Suite Additions

The Conformance Suite SHOULD add test kinds:

```text
EXPERIENCE_MODEL
OPERATIONS_ACTION
OPERATIONS_SUBMISSION
TRACE_GRAPH
COMPONENT_EXPERIENCE_BINDING
```

The existing Conformance Suite schema already owns cross-runtime shared cases and should be extended rather than replaced.

# 9. Processing Model

## 9.1 Artifact Resolution

A processor SHOULD resolve artifacts in the following order:

```text
1. Definition
2. Registry
3. Ontology
4. References
5. Experience
6. Operations
7. Mapping
8. Theme
9. Component
10. Runtime/evidence artifacts
11. Trace Graph generation
```

If a future Package manifest exists, it MAY provide artifact resolution order and variant selection.

## 9.2 Action Invocation

When invoking an Action, a processor MUST:

```text
1. Resolve the Action by ID.
2. Check applicability and actor context when provided.
3. Emit action.started when configured.
4. Evaluate preconditions.
5. Abort if any precondition fails.
6. Obtain confirmation if required.
7. Begin a transaction.
8. Apply effects in declared order.
9. Run the declared validation mode.
10. Produce requested evidence artifacts.
11. Record configured durable Ledger events.
12. Emit action.completed or action.failed.
13. Return an ActionResult.
```

## 9.3 Transaction Semantics

All effects in one Action MUST execute as a single transaction. Intermediate invalid states MUST NOT be externally observed except through explicit diagnostics.

The existing engine already exposes runtime operations and evaluation mechanisms such as `setValue`, `addRepeatInstance`, `removeRepeatInstance`, `getValidationReport`, `evaluateShape`, `getResponse`, diagnostics, replay, and batch evaluation.

## 9.4 Submission Processing

When running a Submission, a processor MUST:

```text
1. Run boundaryValidationMode unless boundaryValidationMode = none.
2. Produce ValidationReport if configured or required.
3. Abort completion when ValidationReport.valid = false and completion is required.
4. Build payload through Mapping (kind: mappingRef) or skip (kind: none).
5. Produce IntakeHandoff if configured.
6. Record RespondentLedger events declared by the invoking Action's recordLedgerEvent effects.
7. Return SubmissionResult.
```

Non-relevant-field handling at the boundary is governed by the active Mapping Document's `targetSchema.nonRelevantBehavior` (see Mapping spec). The Submission does not own NRB policy; it executes a Mapping that does.

# 12. Security Considerations

Operations MUST NOT bypass Definition semantics.

Actions exposed to agents MUST enforce the same preconditions, validation modes, confirmation requirements, posture requirements, and transaction semantics as human-invoked actions.

Signature/custody policy MUST remain in Posture Declaration. Verification evidence MUST remain in Verification Receipt. Operations MAY reference those artifacts but MUST NOT duplicate their internal policy/result structures.

Trace Graphs may expose sensitive relationships among actors, fields, concepts, policies, and workflow artifacts. Processors SHOULD apply the same access controls to Trace Graphs as to the artifacts they index.

---

# 13. Privacy Considerations

Experience Documents may reveal role, task, and process structure. Trace Graphs may reveal policy, validation, and workflow relationships. These artifacts SHOULD be treated as operational metadata and may require access control.

Response data remains governed by the Response schema and host privacy policy. Trace Graphs SHOULD avoid embedding raw Response values unless explicitly generated for a controlled diagnostic context.

---

# 14. Accessibility Considerations

Experience Units SHOULD provide accessibility labels and descriptions where the Unit’s meaning is not obvious from its child items.

Component renderers SHOULD preserve associations among:

```text
Experience Unit
Component node
Definition item
ValidationResult
Action
```

This enables accessible validation summaries, landmark grouping, progress indicators, task-oriented navigation, and assistive agent support.

```json

  "$id": "https://formspec.org/schemas/foundations/1.0",
```

# 18. Final Recommendation

Ship the following additive design:

```text
Add:
  foundations.schema.json   ($defs only — DefinitionRef, TypedRef, HashString, Ref)
  experience.schema.json
  operations.schema.json
  trace-graph.schema.json

Modify:
  component.schema.json     (unitRef, taskRefs, actionRef, conceptRefs; later: $ref foundations DefinitionRef)
  references.schema.json    (typed targets via foundations TypedRef)
  locale.schema.json        ($actor/$task/$unit/$action/$submission namespaces; later: $ref foundations DefinitionRef)
  mapping.schema.json       (nonRelevantBehavior per targetSchema)
  conformance-suite.schema.json

Do not add:
  submission.schema.json
  lifecycle-event.schema.json
  context.schema.json
  projection.schema.json
  evaluation.schema.json
  concepts.schema.json
  signature-policy.schema.json
```

The core ownership model is:

```text
Foundations = shared cross-cutting primitives ($defs only).
Experience  = abstract journey.
Operations  = runtime transitions; orchestrates, does not own artifact bodies.
TraceGraph  = generated relationship index; identity by content digest.
References  = supporting resources and context.
Component   = concrete UI.
Mapping     = boundary translation (including non-relevant-field behavior).
```

This preserves the spirit of XForms, XIML, and UsiXML/CAMELEON while staying faithful to the Formspec schemas that already exist. Foundations extraction is the longest-leverage architectural move: every future companion spec inherits the primitives instead of re-litigating them.
